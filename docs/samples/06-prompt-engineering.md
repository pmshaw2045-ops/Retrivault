# Prompt Engineering 最佳实践

## RAG 系统的 Prompt 设计

在 [[RAG 基础知识]] 系统中，Prompt 是最容易被忽略的环节。同样的检索结果，不同的 Prompt 可导致回答质量天差地别。

### System Prompt 模板

```markdown
你是个人知识库助手。你的回答严格基于以下检索到的知识片段。
如果知识片段不包含回答所需信息，直接说"没有相关信息"，
绝不编造。

## 回答规则
1. 用自然的中文回答用户问题
2. 在答案中使用 [N] 标注引用来源
3. 多个片段涉及同一观点时引用所有编号
4. 不引入外部知识
5. 知识片段中的 [[xxx]] 保留为纯文本
```

> [!warning] 关键点
> "绝不编造"这条指令极其重要。没有这条，LLM 会自行发挥，破坏 RAG 的可信度。

### Few-shot 示例的价值

对于 Query 改写任务，Few-shot 示例比 System Prompt 本身更重要：

```
输入：怎么给新员工解释Agent概念
输出：Agent 概念 新员工 解释

输入：Agent落地有哪些关键原则  
输出：Agent 落地 关键 原则
```

如 [[Embedding 模型选型指南]] 所述，改写后的查询比原始自然语言查询检索效果更好。

## Prompt 优化的 5 个原则

### 1. 角色设定明确
告诉 LLM 它是什么角色，比说"你是助手"效果好 3 倍。

### 2. 约束在前
"你不做什么"比"你要做什么"优先级更高。先画红线再给自由。

### 3. 格式示例
给出期望的输出格式示例，LLM 就能稳定产出该格式。

### 4. 温度调低
RAG 场景推荐 0.1-0.3。温度越低，越稳定，幻觉越少。

### 5. Token 预算
给 LLM 留足够的 token 让它"思考"。通常 system + user prompt 不超过 6K tokens。

## 常见陷阱

| 陷阱 | 表现 | 对策 |
|------|------|------|
| Prompt 太长 | LLM 忽略中间内容 | 精简，关键放在开头和结尾 |
| 指令矛盾 | "简洁"+"详细分析" | 一次只说一个要求 |
| 角色混乱 | 既是助手又是专家 | 统一一个角色 |
