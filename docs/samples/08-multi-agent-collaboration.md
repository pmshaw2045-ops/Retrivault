# 多 Agent 协作系统设计

单个 Agent 能力有限，多 Agent 协作通过分工、通信和协调解决复杂任务。

## 核心架构模式

### 1. 编排器模式（Orchestrator）

一个中央调度 Agent（编排器）负责任务分解、分配和结果汇总。子 Agent 各自专注特定领域。

```
用户输入
    │
    ▼
┌──────────┐
│ 编排器    │ ← 任务理解、分解、协调
└────┬─────┘
     │
     ├──► 研究 Agent ──► 信息检索与汇总
     ├──► 写作 Agent ──► 内容生成与润色
     ├──► 审查 Agent ──► 质量检查与修正
     └──► 代码 Agent ──► 代码编写与调试
```

优势：结构清晰，易于监控。劣势：编排器成为瓶颈和单点故障。

### 2. 对话式模式（Conversational）

Agent 之间自由对话，通过自然语言交换信息。典型实现如 AutoGen 的对话组（Chat Group）：

> 每个 Agent 有自己的角色描述和发言规则，在共享对话上下文中轮流发言，直到达成共识或完成任务。

优势：灵活、适应性强。劣势：对话可能发散，收敛性无保障。

### 3. 市场模式（Market）

Agent 通过发布和竞标任务协作，类似自由市场机制。

| 角色 | 职责 |
|------|------|
| 任务发布者 | 描述需求、设定预算、评估方案 |
| 竞标者 | 评估自身能力、报价、执行任务 |
| 仲裁者 | 评估结果、解决争议、支付报酬 |

## 关键技术挑战

### 通信协议

Agent 间通信需要结构化格式以确保互操作性。常用方案：

- **JSON-RPC**：远程过程调用，适合请求-响应模式
- **事件驱动**：发布-订阅模式，适合异步通知
- **共享黑板**：公共存储区，Agent 读写共享信息

```python
# 简化的 Agent 通信消息格式
@dataclass
class AgentMessage:
    sender: str           # 发送方 Agent ID
    receiver: str         # 接收方 Agent ID（broadcast 为 "*"）
    msg_type: str         # request / response / broadcast / error
    task_id: str          # 关联的任务 ID
    payload: dict         # 消息内容
    timestamp: float      # 时间戳
```

### 记忆共享

多 Agent 系统的记忆分为三级：

1. **个体记忆**：每个 Agent 独立的对话历史和工作状态
2. **共享记忆**：所有 Agent 可读的公共知识库（如向量数据库）
3. **全局状态**：系统级的任务进度、资源分配和健康监控

### 冲突解决

Agent 间产生冲突（如对同一问题给出不同答案）时，常见解决策略：

- **投票机制**：多个 Agent 独立判断，取多数意见
- **分层裁决**：上级 Agent 裁定下级分歧
- **置信度对比**：比较各自置信度，取置信度高者
- **外部验证**：调用外部工具/API 验证事实

## 典型应用场景

### 软件开发

| Agent 角色 | 职责 |
|-----------|------|
| 产品经理 Agent | 需求分析、用户故事编写、验收标准定义 |
| 架构师 Agent | 技术选型、模块划分、接口设计 |
| 开发 Agent | 代码实现、单元测试、重构 |
| QA Agent | 测试用例编写、自动化测试、缺陷报告 |
| DevOps Agent | CI/CD 配置、部署监控、告警处理 |

### 研究分析

研究场景下，多个 Agent 可并行调研不同方向，汇总交叉验证结论。例如：研究 Agent A 搜索学术论文，Agent B 检索技术博客，Agent C 分析竞品文档，最后由合成 Agent 整合为综合报告。

### 客户支持

分层 Agent 架构：
- **L1 客服 Agent**：处理常见问题（FAQ 匹配、订单查询），解决率约 80%
- **L2 专家 Agent**：处理复杂问题（技术故障、投诉升级）
- **L3 人类交接**：Agent 无法处理时，整理上下文转接人工
